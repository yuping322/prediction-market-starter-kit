# 套利机器人技术设计文档

## 1. 文档目标

本文面向当前 `prediction-market-starter-kit` 的扩展设计，目标是在 Polymarket 二元市场与短周期市场中构建一套可落地的套利机器人。系统核心由六个数学模型组成：`Bayesian + Edge + Spread + Stoikov + Kelly + Monte Carlo`。

该系统的核心目标不是“预测一切”，而是：

1. 用实时数据快速修正市场判断。
2. 识别真正具有正期望收益的套利机会。
3. 在流动性有限、成交不完整、延迟存在的现实条件下完成双腿组装。
4. 控制仓位、库存风险与最大回撤。
5. 让回测结果尽可能接近实盘表现。

---

## 2. 设计范围与假设

### 2.1 适用场景

- `Polymarket` 二元市场（`YES/NO`）
- 短周期事件市场，尤其是 `5m` 到期市场
- 单市场双边套利、相关市场统计套利、订单流不平衡套利
- 以限价单为主、必要时使用吃单补腿

### 2.2 关键现实约束

套利机器人在设计时必须显式建模以下成本与不确定性：

- 手续费
- 滑点
- 网络延迟
- 部分成交
- 双腿成交不同步
- 流动性枯竭
- 概率估计误差
- 连续失败交易集群

### 2.3 设计原则

- 先判断“是否真的有优势”，再考虑“如何下单”。
- 所有信号必须折算为统一的 `Expected Value (EV)`。
- 所有执行策略必须服从库存约束与风险预算。
- 所有回测假设必须可被实盘数据校准。

---

## 3. 系统总体架构

```text
市场数据流
  -> 特征提取
  -> Bayesian 概率更新
  -> Edge 优势评估
  -> Spread 定价与配对结构
  -> Stoikov 库存修正
  -> Kelly 仓位计算
  -> Monte Carlo 风险门控
  -> 执行引擎
  -> 订单/仓位/风控监控
```

### 3.1 模块分层

1. **数据层**
   - 盘口快照
   - 成交流
   - 历史价差序列
   - 订单状态与成交回报

2. **信号层**
   - Bayesian：更新事件概率与市场状态
   - Edge：判断是否存在正优势
   - Spread：判断价差是否偏离常态，并给出挂单位置

3. **执行层**
   - Stoikov：根据库存调整报价中心
   - Kelly：根据优势与赔率决定仓位大小
   - 执行器：下单、补腿、撤单、超时处理

4. **风险层**
   - Monte Carlo：离线校验策略稳健性
   - 在线风控：限额、止损、最大回撤、熔断

## 4. 六个核心数学模型

## 4.1 Bayesian：实时修正市场认知

### 4.1.1 目标

Bayesian 模型用于根据新到达的数据快速更新市场状态判断，例如：

- 当前是趋势行情还是震荡行情
- 下一小段时间价格上涨的概率是多少
- 当前异常订单流是否代表信息优势

### 4.1.2 数学表达

设：

- `S`：市场状态，例如 `trend_up`、`trend_down`、`range`
- `D`：最新观测数据，例如巨量买单、盘口失衡、成交量突增

则后验概率为：

\[
P(S \mid D) = \frac{P(D \mid S) P(S)}{P(D)}
\]

如果状态空间为多个离散状态，则：

\[
P(S_i \mid D) = \frac{P(D \mid S_i) P(S_i)}{\sum_j P(D \mid S_j) P(S_j)}
\]

### 4.1.3 输入特征

- `Bid/Ask` 深度变化
- 最近 `10ms / 100ms / 1s` 成交量失衡
- 取消单速度
- 微价格偏移
- 价格跳变频率
- 与相关市场的价差偏移

### 4.1.4 输出

- `p_up`：下一短时窗口上涨概率
- `p_down`：下跌概率
- `regime`：市场状态分类
- `confidence`：后验置信度

### 4.1.5 业务解释

例如：

- 过去 `5` 分钟统计下，价格上涨概率约为 `50%`
- 过去 `10ms` 内买盘成交量显著大于卖盘，且 `Bid_1` 出现异常大单
- Bayesian 更新后，`p_up` 从 `50%` 上修到 `85%`

此时模型并不直接下单，而是将更新后的概率交给 `Edge` 模型，判断这是否足以形成可兑现的数学优势。

## 4.2 Edge：判断错位是否具有真实优势

### 4.2.1 目标

`Edge` 模型用于区分：

- 只是“看起来有价差”
- 还是“扣除成本后依然有正期望收益”

只有 `EV > 0` 的机会，才值得进入执行阶段。

### 4.2.2 基本定义

定义：

\[
Edge = EV = \mathbb{E}[Payoff] - Costs
\]

其中：

- `Costs` 包括手续费、滑点、延迟损耗、补腿失败成本
- `Payoff` 包括套利结构到收敛时的理论收益

### 4.2.3 二元市场双边套利

在 Polymarket 二元市场中，若两边都可以买入，且：

\[
p_{yes} + p_{no} < 1
\]

则存在静态套利空间。净优势可写为：

\[
Edge = 1 - (p_{yes} + p_{no}) - c
\]

其中 `c` 为总成本。

例：

- `p_yes = 0.47`
- `p_no = 0.48`
- `c = 0.02`

则：

\[
Edge = 1 - 0.95 - 0.02 = 0.03
\]

即净优势为 `3%`。

### 4.2.4 订单流 Edge

通过盘口微观结构可以定义短线优势。常用微价格为：

\[
MicroPrice = \frac{Ask \cdot Q_{bid} + Bid \cdot Q_{ask}}{Q_{bid} + Q_{ask}}
\]

其中：

- `Bid`、`Ask` 为最优买卖价
- `Q_bid`、`Q_ask` 为对应挂单深度

若买盘深度远大于卖盘，则 `MicroPrice` 会高于当前中间价，说明短线向上压力更大。这种买卖力量的非对称性可被定义为一种可兑现的 `Edge`。

### 4.2.5 Edge 类型

系统需支持至少三类机会：

1. **空间套利**：同一事件 `YES + NO < 1`
2. **统计套利**：相关市场价差偏离历史均值
3. **订单流套利**：盘口失衡、队列坍塌、微价格偏移

### 4.2.6 输出

- `edge_value`
- `edge_type`
- `expected_holding_time`
- `execution_confidence`

## 4.3 Spread：决定价差是否值得交易，以及挂单放在哪里

### 4.3.1 目标

`Spread` 模型负责两件事：

1. 识别当前价差是否超出正常噪声范围
2. 给出最优挂单位置与对冲结构

### 4.3.2 基本价差定义

盘口价差：

\[
Spread_t = Ask_t - Bid_t
\]

围绕中间价 `m` 的基础挂单可定义为：

\[
Bid^* = m - \delta
\]

\[
Ask^* = m + \delta
\]

其中：

- `m` 为当前中间价
- `\delta` 为模型给出的偏移量

### 4.3.3 跨市场价差定义

对于两个相关市场 `A` 和 `B`，定义价差：

\[
S_t = P_t^A - \beta P_t^B
\]

其中：

- `P_t^A`、`P_t^B` 为两个市场的价格
- `\beta` 为对冲系数；若简化处理，可先取 `1`

### 4.3.4 Z-Score 判断异常程度

为了判断当前价差是否为噪声，定义：

\[
Z_t = \frac{S_t - \mu_S}{\sigma_S}
\]

其中：

- `S_t`：即时价差
- `\mu_S`：过去 `N` 个周期的平均价差
- `\sigma_S`：过去 `N` 个周期的价差标准差

### 4.3.5 决策阈值

- 当 `|Z| >= 1`：进入警戒区，开始跟踪
- 当 `|Z| >= 2`：触发均值回归交易
- 当 `|Z| < 1`：视为回归完成，择机止盈或退出

### 4.3.6 输出

- `spread_value`
- `z_score`
- `target_bid`
- `target_ask`
- `hedge_ratio`

## 4.4 Stoikov：在有限流动性中完成库存管理与报价偏移

### 4.4.1 目标

套利真正困难的地方，往往不是发现错位，而是及时组装结构的两个部分，同时避免持仓越积越偏。

传统做市策略围绕中间价对称挂单；`Stoikov` 模型则认为：当你已经持有库存时，你的“保留价格”必须偏移。

### 4.4.2 保留价格公式

\[
r_t = s_t - q_t \gamma \sigma_t^2 (T - t)
\]

其中：

- `s_t`：当前市场中间价
- `q_t`：净持仓量；多头过重时为正
- `\gamma`：风险厌恶系数
- `\sigma_t`：市场波动率
- `T - t`：距离收盘或窗口结束的剩余时间

### 4.4.3 业务解释

若机器人持有过多 `YES` 头寸，则 `q_t > 0`，模型会使：

\[
r_t < s_t
\]

这意味着：

- 卖价会更积极，以便尽快出货
- 买价会更保守，避免继续加仓

换言之，`Stoikov` 的作用是：

- 在库存失衡时主动偏移报价中心
- 用更现实的方式完成双腿组装
- 在短寿命市场中减少“有一条腿、没另一条腿”的暴露时间

### 4.4.4 实际输出

- `reservation_price`
- `inventory_skew`
- `bid_adjustment`
- `ask_adjustment`

---

## 4.5 Kelly：决定仓位大小，而不是决定方向

### 4.5.1 目标

`Kelly` 模型用于根据优势大小、胜率与赔率来决定应该投入多少资金。它不负责告诉你买不买，而负责告诉你“买多少”。

### 4.5.2 基本公式

设：

- `p`：胜率
- `q = 1 - p`
- `b`：赔率，即赢时净赚与本金的比值

则 Kelly 最优仓位比例为：

\[
f^* = \frac{bp - q}{b}
\]

### 4.5.3 二元合约的赔率换算

若 `YES` 价格为 `c`，则：

- 买入 `1` 单位的成本为 `c`
- 若事件发生，最终兑付为 `1`
- 净利润为 `1 - c`

因此赔率可写为：

\[
b = \frac{1 - c}{c}
\]

### 4.5.4 示例

若：

- `c = 0.40`
- Bayesian 给出 `p = 0.55`

则：

\[
b = \frac{1 - 0.40}{0.40} = 1.5
\]

\[
f^* = \frac{1.5 \times 0.55 - 0.45}{1.5} = 0.25
\]

即理论上可投入总资产的 `25%`。

### 4.5.5 实盘改造

实盘中不建议使用满 Kelly，而应使用分数 Kelly：

\[
f_{live} = \lambda f^*, \quad \lambda \in [0.1, 0.5]
\]

并叠加以下约束：

- 单市场最大敞口
- 单事件最大敞口
- 单次交易最大下单金额
- 剩余流动性容量
- 未对冲库存上限

### 4.5.6 输出

- `target_notional`
- `capital_fraction`
- `size_cap_reason`

---

## 4.6 Monte Carlo：在部署前验证策略是否扛得住真实世界

### 4.6.1 目标

`Monte Carlo` 用历史数据与随机扰动对策略进行大量模拟，以评估：

- 资金曲线稳定性
- 最大回撤 (`MDD`)
- 连续亏损时的承受能力
- 参数误差对结果的敏感性

### 4.6.2 模拟维度

每一条模拟路径都应随机采样以下变量：

- 不同成交率
- 不同滑点水平
- 随机执行延迟
- 双腿不完整结构比例
- Bayesian 概率估计误差
- 连续失败交易集群
- 不同波动 regime 切换

### 4.6.3 输出指标

- `Expected PnL`
- `Max Drawdown`
- `PnL distribution`
- `Ruin Probability`
- `Hit Rate`
- `Inventory Tail Risk`

### 4.6.4 通过标准示例

上线前可设置如下风险门槛：

- `10,000` 次随机路径下，策略整体期望收益为正
- `95%` 分位下最大回撤不超过阈值
- 连续失败交易集群下不触发爆仓或流动性枯竭
- 参数扰动后结论仍然稳健

## 5. 六模型协同下的完整交易流程

### 5.1 交易流程

1. **Bayesian** 根据最新订单流更新下一短时窗口涨跌概率与市场状态。
2. **Edge** 评估当前错位在扣除手续费、滑点与延迟后是否仍为正优势。
3. **Spread** 计算价差是否偏离历史常态，并给出挂单位置：`m - δ` 买入、`m + δ` 卖出。
4. **Stoikov** 如果库存失衡，则调整报价中心，优先完成减仓与补腿。
5. **Kelly** 根据胜率与赔率分配仓位，限制过度下注。
6. **Monte Carlo** 作为上线前与日常再校准的风险基准，确保最大回撤与尾部风险在可接受区间。
7. **执行引擎** 下发订单、监听回报、必要时做主动补腿或撤单重挂。

### 5.2 伪代码

```text
onMarketEvent(event):
  features = extractFeatures(event, orderbook, trades)
  posterior = bayesian.update(features)
  edge = edgeModel.evaluate(posterior, orderbook, costs)

  if edge.ev <= 0:
    return

  quote = spreadModel.quote(orderbook, relatedMarkets)
  adjustedQuote = stoikov.adjust(quote, inventory, volatility, timeToExpiry)
  size = kelly.size(posterior.winProb, adjustedQuote.odds, bankroll, limits)

  if not riskGate.pass(edge, size, inventory):
    return

  execution.place(adjustedQuote, size)
```

## 6. 回测与实盘如何对齐

这是该系统最关键的工程问题之一。套利策略最容易出现的问题不是“公式不对”，而是“回测赢、实盘亏”。因此设计上必须从一开始就把回测与实盘统一。

## 6.1 核心原则

### 原则 1：使用同一套事件驱动时钟

不能只用 `1s` 或 `1m` K 线回测微观套利。回测和实盘都应基于：

- 盘口快照
- 成交事件
- 订单状态变更
- 真实时间戳

### 原则 2：使用同一套成本模型

回测中的手续费、滑点、延迟、补腿失败概率，必须与实盘统计口径一致，并按日/周校准。

### 原则 3：使用同一套成交模型

如果实盘主要靠限价挂单成交，则回测必须模拟：

- 排队位置
- 队列前撤单
- 对手单冲击
- 部分成交
- 超时撤单

不能用“价格到达即全部成交”的理想假设。

### 原则 4：信号与执行分开评估

回测时需要将：

- 信号质量
- 执行损耗
- 库存风险

分别归因，否则无法知道问题出在模型、成交、还是风险管理。

## 6.2 建议的对齐框架

| 维度 | 回测要求 | 实盘要求 |
| --- | --- | --- |
| 数据源 | 使用逐笔成交 + 盘口重建 | 保存原始事件流 |
| 时间戳 | 保留毫秒/微秒级时间 | 记录接收、决策、发单、成交时间 |
| 手续费 | 固定 + 分层费率建模 | 用真实成交账单回填 |
| 滑点 | 按盘口深度模拟 | 按成交回报统计 |
| 延迟 | 注入随机网络与撮合延迟 | 实测端到端延迟分布 |
| 成交率 | 基于排队位置与冲击建模 | 按订单级别统计成交率 |
| 概率误差 | 对 Bayesian 输出做扰动测试 | 比较预测概率与真实命中率 |
| 仓位限制 | 强制执行风控约束 | 与线上风控口径一致 |

## 6.3 上线流程建议

1. **历史回放**：在订单级数据上回放策略。
2. **参数冻结**：冻结一组参数，不允许边跑边调。
3. **纸面交易**：只生成订单，不真实成交，先验证信号与执行逻辑。
4. **影子实盘**：真实跟踪盘口，记录若真实下单会发生什么。
5. **小资金上线**：仅用极小仓位验证成交模型。
6. **持续校准**：每天更新滑点、延迟、成交率、补腿成功率分布。

## 7. 风控设计

### 7.1 预交易风控

- `EV <= 0` 禁止交易
- `edge_value < edge_threshold` 禁止交易
- `|Z| < z_enter` 禁止统计套利入场
- 单次交易名义金额不得超过上限
- 单市场库存不得超过上限

### 7.2 交易中风控

- 双腿未完成超过设定时长，立即补腿或强平
- 市场数据陈旧时停止挂单
- 波动率突增时扩大价差或暂停策略
- 网络异常、订单回报异常时触发熔断

### 7.3 组合级风控

- 日内最大亏损限制
- 最大回撤限制
- 相关市场总敞口限制
- 单一事件集中度限制
- 极端尾部事件人工接管开关

## 8. 推荐的工程落地方案

考虑到当前仓库本质上是一个 `Next.js + Polymarket` 的前端/交易应用，建议将套利机器人拆为两个部分：

### 8.1 保留在当前仓库中的内容

- 参数配置界面
- 风控面板
- 订单与成交监控页面
- 回测结果展示
- 人工启停开关

### 8.2 独立进程/服务承载的内容

- 高频数据订阅
- 特征提取
- 六模型计算
- 订单执行循环
- 实时风险控制

原因是：

- `Next.js` 更适合 UI、配置与运营后台
- 高频套利逻辑更适合独立常驻进程
- 可避免请求生命周期、冷启动、部署形态对实时性的影响

### 8.3 推荐模块划分

```text
lib/strategy/
  features.ts
  costs.ts
  risk-gate.ts
  execution.ts
  models/
    bayesian.ts
    edge.ts
    spread.ts
    stoikov.ts
    kelly.ts
    monte-carlo.ts

app/api/arb/
  config/route.ts
  status/route.ts
  reports/route.ts
```

如果后续进入真正高频或准高频阶段，建议进一步拆成独立 worker：

```text
services/arb-engine/
  market-data.ts
  signal-engine.ts
  execution-engine.ts
  position-manager.ts
  calibration.ts
```

## 9. 参数建议与默认值方向

以下不是最终参数，只是工程起步时的默认方向：

- `Bayesian` 更新窗口：`10ms / 100ms / 1s`
- `Edge` 最小阈值：需覆盖手续费 + 预期滑点 + 延迟损耗 + 安全边际
- `Spread` 统计窗口：`N = 100 ~ 1000` 个事件点
- `Z-Score` 入场阈值：`2.0`
- `Z-Score` 退出阈值：`0.5 ~ 1.0`
- `Stoikov γ`：从保守值开始，小资金下逐步标定
- `Kelly`：使用 `0.1x ~ 0.25x` 分数 Kelly
- `Monte Carlo`：至少 `10,000` 条路径

---

## 10. 结论

这套套利机器人不是单一预测模型，而是一个由六个模型耦合而成的决策系统：

- `Bayesian` 负责更新认知
- `Edge` 负责确认优势
- `Spread` 负责发现并定价错位
- `Stoikov` 负责库存约束下的动态报价
- `Kelly` 负责仓位分配
- `Monte Carlo` 负责部署前后的风险验证

如果把它们串成统一流程，本质上是在回答六个问题：

1. 市场现在最可能是什么状态？
2. 这个错位扣除成本后还有没有正收益？
3. 应该在哪个价格挂单？
4. 当前库存下报价应不应该偏移？
5. 这笔交易该下多大？
6. 如果连续做一万次，这套系统会不会把自己打穿？

只有这六个问题同时被回答，套利机器人才是一个可上线的系统，而不是停留在纸面上的策略想法。
